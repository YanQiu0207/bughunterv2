# Roadmap（里程碑与渐进规划）

> 渐进式、每步可用可验证。当前路线已从「设计优先」进入「原型验收优先」。

## 当前状态快照

- M0/M1/M2/M3A 已完成原型实现。
- 本地非集成测试已通过；真实 LLM、真实 Java 项目、真实 SVN 工作副本仍需手工验收。
- 下一步不继续扩展能力，先按 `docs/design-docs/bugfix-agent/pipeline/e2e-manual-acceptance.md` 跑通端到端验收。

## M0 — 设计与地基（已完成）

- [x] 澄清核心流程需求（spec 1–3 章）
- [x] 确立项目定位（两大块）与文档体系
- [x] 确认 MVP 范围、诊断可信度标准、文档体系、隔离方向（已在 ADR-0001 固化）
- [x] 系统设计（spec 第 4 章：骨架 + 嫌疑变量策略 + 码内/码外判定 + 上下文控制 + 落盘格式）
- [x] 定义工具接口契约（`docs/standards/tool-contract.md`，已补齐）

## M1 — 诊断 MVP（纯代码分析，已实现，待真实 LLM 验收）

- [x] 工具：`read_source` + `find_callers`（纯静态代码分析，无外部数据访问）
- [x] 跑通 F1 诊断：输入堆栈 + 本地源码目录 → 输出「根因结论 + 证据链 + 置信度」
- [x] 碰到码外来源时如实标注「无法验证，置信度 low」，不暂停
- [ ] 用 canonical example（NPE 最简例子）在真实 LLM 配置下验收

## M2 — 修复与编译反馈环（已实现，待真实 Java 项目验收）

- [x] F2 生成修复方案 / F3 应用到隔离工作区 / F4 编译报错回灌迭代
- [x] 跑通「方案 → review → 改 → 编译 + 单测 → 再改」的人在环循环
- [x] 验证标准：编译通过 + 单测全绿（运行时验证交 QA）
- [x] 隔离策略：已从 Linux 硬链接工作区调整为干净 SVN 缓存副本方案（见 ADR-0004）
- [x] 同步实现 `apply_fix.py`，从 `workspace/cache/svn-clean` 复制出 `workspace/fix/<fix_id>/`
- [ ] 在最小 Java 项目上验收 `fix.py`

## M3A — SVN 写回与提交链路（已实现，待真实 SVN 验收）

- [x] `commit_fix.py` 读取 verified `FixProposal`
- [x] `svn update` 后做冲突检测
- [x] 支持 `--dry-run` 先展示 diff，不写文件
- [x] 需要 `--yes` 才写回并提交，保留人工 gate
- [ ] 在最小 SVN 工作副本上验收无冲突、冲突、失败回滚路径

## M3B — 本地复现能力（下一阶段）

- 引入 `start_local_server` + `query_local_db`：启动本地服务、在本地 DB 验证数据假设
- 升级 F1 诊断：码外来源可在本地环境取证，提升置信度

## M4 — 测试辅助与版本纳入增强

- F5 测试辅助 / F6 工作区提交合回（SVN 半自动）
- 在 M3A 基础上增强真实业务项目的验收、回滚和审计体验

## M5 — 对接线上日志

- 引入 `search_logs`：接入线上日志，进一步提升码外来源的取证能力
- **永久约束**：线上 DB 不纳入工具范围

## 横向持续项（块 B，贯穿全程）

- 规范 / 标准 / 文档 / ADR 的持续沉淀
- agent 协作机制（提示词、子 agent 分工、上下文策略）的迭代
