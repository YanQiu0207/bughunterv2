# Roadmap（里程碑与渐进规划）

> 渐进式、每步可用可验证。前期以「把需求/设计/地基打好」为重，不急于编码。
> 下列里程碑为**初稿提议**，待与老板讨论后定稿。

## M0 — 设计与地基（进行中）

- [x] 澄清核心流程需求（spec 1–3 章）
- [x] 确立项目定位（两大块）与文档体系
- [x] 确认 MVP 范围、诊断可信度标准、文档体系、隔离方向（已在 ADR-0001 固化）
- [x] 系统设计（spec 第 4 章：骨架 + 嫌疑变量策略 + 码内/码外判定 + 上下文控制 + 落盘格式）
- [x] 定义工具接口契约（`docs/standards/tool-contract.md`，已补齐）

## M1 — 诊断 MVP（纯代码分析）

- 工具：`read_source` + `find_callers`（纯静态代码分析，无外部数据访问）
- 跑通 F1 诊断：输入堆栈 + 本地源码目录 → 输出「根因结论 + 证据链 + 置信度」
- 碰到码外来源时如实标注「无法验证，置信度 low」，不暂停
- 用 canonical example（NPE 最简例子）验证骨架

## M2 — 修复与编译反馈环

- F2 生成修复方案 / F3 应用到隔离工作区 / F4 编译报错回灌迭代
- 跑通「方案 → review → 改 → 编译 + 单测 → 再改」的人在环循环
- 验证标准：编译通过 + 单测全绿（运行时验证交 QA）
- 隔离策略：Linux 硬链接工作区，原目录全程不动（见 ADR-0003）

## M3 — 本地复现能力

- 引入 `start_local_server` + `query_local_db`：启动本地服务、在本地 DB 验证数据假设
- 升级 F1 诊断：码外来源可在本地环境取证，提升置信度

## M4 — 验证与版本纳入

- F5 测试辅助 / F6 工作区提交合回（SVN 半自动）
- 打通到「纳入版本管理」的完整链路

## M5 — 对接线上日志

- 引入 `search_logs`：接入线上日志，进一步提升码外来源的取证能力
- **永久约束**：线上 DB 不纳入工具范围

## 横向持续项（块 B，贯穿全程）

- 规范 / 标准 / 文档 / ADR 的持续沉淀
- agent 协作机制（提示词、子 agent 分工、上下文策略）的迭代
