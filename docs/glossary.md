# 术语表（Glossary）

> 本项目统一概念词典。同一概念全篇只用一个说法；新增术语先写这里，再用于其他文档。
> 按首字母排序，括号内附英文对照。

---

## 抛出点（Exception Throw Point）

异常堆栈**栈顶**所指向的代码位置——即「异常从哪行被抛出」。

**不等于根因点**。抛出点是症状显现处，根因可能在调用链更深处或代码之外。
参见：→ 根因点、→ 根因。

---

## 根因（Root Cause）

导致 bug 的**最终原因**——「坏值或错误逻辑真正产生的地方与原因」。

根因可能在代码内（写死的 null、错误的逻辑分支），也可能在代码外（DB 里一条坏数据、接口返回异常值）。
参见：→ 抛出点、→ 根因点、→ 回溯循环。

---

## 根因点（Root Cause Point）

调用链中**坏值或错误逻辑真正产生的代码位置**（或外部来源）。

与抛出点的区别：抛出点是「症状爆发点」，根因点是「病灶所在」。
示例：抛出点在 `printName()` 里的 `name.length()`，根因点在调用方 `handle()` 传入了 `null`。

---

## 回溯循环（Backtrace Loop）

F1 诊断的**核心执行骨架**，定义如下：

```
定位嫌疑变量
  → 回溯其来源
      ├─ 来源在代码内   → 继续读码，顺数据流上溯
      └─ 来源在代码外   → 调工具取证（查 DB / 日志 / 接口 / 反编译）
  → 直到坐实根因，或线索耗尽
  → 输出证据链
```

简单 bug 是退化情况（不触发工具），复杂 bug 是完整情况（触发工具取证），两者复用同一骨架。
参见：→ 嫌疑变量、→ 码内 / 码外、→ 证据链。

---

## 嫌疑变量（Suspect Variable）

在回溯过程中被锁定的、**可能携带坏值的变量**。

诊断的起点是在抛出点或附近识别第一个嫌疑变量，然后沿数据流向上回溯其来源，直到找到根因。

---

## 码内 / 码外（In-Code / Out-of-Code）

在回溯循环中判定变量来源的二分：

- **码内**：变量的值由源码内的逻辑产生（字面量、运算、另一个变量赋值等） → 继续读码回溯。
- **码外**：变量的值来自外部系统（数据库查询、HTTP 接口、配置文件、JVM 环境等） → 调工具取证。

是决定「继续读码」还是「调工具」的关键判定点。

---

## 工具契约（Tool Contract）

外部能力的**接口定义**——规定工具接受哪些参数、返回哪种结构、有哪些约束（如返回条数上限）。

Mock 实现与真实实现共同遵守同一契约，因此可以**热替换**，不改调用方代码。
规范详见 `docs/standards/tool-contract.md`（待建）。
参见：→ mock 工具、→ 热替换。

---

## mock 工具（Mock Tool）

遵循「→ 工具契约」的**占位实现**，用预设的固定数据模拟真实工具的响应。

用途：在真实环境未对接前，让诊断流程可以端到端跑通。
约束：mock 数据必须覆盖「正常返回」和「空结果 / 异常返回」两类场景。

---

## 热替换（Hot Swap）

在不修改调用方代码的前提下，将 mock 工具替换为真实工具实现的操作。

前提：mock 与真实实现严格遵守同一「→ 工具契约」。

---

## 人在环（HITL, Human In The Loop）

流程中设置**人工确认节点**的协作模式——agent 推进到关键决策点时，输出结果供人 review，人确认后再继续。

本项目的人工 gate 点包括：根因 review、修复方案 review、编译触发、SVN 提交确认。

---

## 证据链（Evidence Chain）

F1 诊断的**核心输出结构**，包含四个字段：

| 字段 | 含义 |
|------|------|
| 根因假设 | 一句可证伪的原因陈述（「A 是 null 因为 B 传入了 null」） |
| 支撑证据 | 具体到「哪行代码 / 哪条数据」的事实列表 |
| 反证检查 | 主动排除其他可能、确认假设唯一成立 |
| 修复方向 | 修复思路（具体改法属 F2，不在 F1 输出） |

「可证伪」要求：假设必须是能被反驳的具体陈述，不接受「可能是数据问题」这类模糊表述。

---

## 诊断可信度（Diagnostic Confidence）

对根因假设**可靠性的评估**，不是布尔值（对/错），而是一个有依据的评级。

评级依据：支撑证据的具体程度、反证检查是否覆盖主要替代假设、线索是否被耗尽。
Agent 输出根因结论时，必须附上可信度评级及其理由；不允许输出无评级的结论。

---

## 隔离工作区（Isolated Workspace）

从目标项目主分支拉出的**独立目录副本**，作为 agent 代码改动的落地区。

目的：agent 的所有文件修改发生在此处，不污染主工作副本，人工确认后再合回。
细节设计留入 M2 启动前（见 ADR-0001 开放问题）。
