# 文档撰写规范（Documentation Convention）

> 本项目靠 agent 长期开发，文档是「人 + agent 的共享长期记忆」。一段有歧义的文字，会让后续接手者（哪怕换个会话的同一个 agent）误解，或被迫重新推导。**故文档质量是地基，全项目遵守。**

## 四条质量要求

1. **精确**：用词准确；同一概念全篇用同一说法；未定的点明确标「待定」，不用「大概 / 可能」掩盖。
2. **无歧义**：任一句话不留第二种合理解读；消除指代不清的「它 / 这个 / 上述」，必要时点名具体对象。
3. **引导性（可执行）**：读者读完应知道「接下来该怎么做」，而非只知道「有这么回事」。
4. **自包含 + 例子**：单看这一段（脱离当时对话）就能懂；抽象的点用具体例子锚定。

## TODO / 待办的写法

每条待办简明扼要，但信息要完整到「可直接上手」，包含：

- **做什么**：一句话目标。
- **现状 / 为什么**：当前情况、为何要改。
- **怎么做 / 完成标准**：具体动作，或「做成什么样算完成」。
- **（可选）料 / 例子**：已有的参考、输入，或一个示例。

### 反例 → 正例（对照）

❌ **反例**（有歧义、不可执行）：

> 收紧两处过强措辞：「自检有数学上限」「优化可信度而非真值」。

问题：没说「为什么过强」「收紧成什么样」，后续看到还得重新推导。

✅ **正例**（精确、带改法与理由）：

> 「自检有数学上限」→ 改为「自我验证的可靠性有限」。
> 理由：「数学上限」暗示已被证明的定理，实为经验性局限，属夸大。

## 适用范围

全项目所有文档：spec、ADR、standards、roadmap、TODO、代码注释等。
**新增或修改任何文档前，对照本规范自检。**
