# ADR-0001：项目地基决策

**状态**: 已接受（部分条目标注「待定」）
**日期**: 2026-06-19

## 背景

需求澄清阶段结束，在正式进入 M1 诊断设计之前，把「不会再改变」的地基共识固化在此，避免后续会话重复推导或产生隐式假设。

## 决策

### 1. 项目定位与边界

- **bughunterv2** 是全新的 **Python 3 工具**，物理上独立于它所分析的目标项目（Java + Maven + SVN）。
- bughunterv2 只**读取 / 分析**目标项目的源码与产出；修复改动写入「隔离工作区」，不直接写入目标项目的主工作副本。

### 2. 两大块并行演进

- **块 A — 核心流程**：诊断 → 修复 → 验证 → 纳入版本（F1–F6）的工程实现。
- **块 B — 方法论沉淀**：规范、标准、文档、agent 协作机制的持续整理。
- 二者缺一不可，全程并行，不因「先做 A 再做 B」或「先做 B 再做 A」而偏废。

### 3. 核心流程形态（前期）

- F1–F6 各阶段设计为**独立可调用接口**，每阶段产出落盘，由人工串联；不强求自动编排。
- 自动化编排属 M3/M4 以后的演进，前期不提前引入。

### 4. 工具策略：mock 优先 + 统一契约

- 所有外部能力（DB 查询、日志、接口调用、反编译、SVN、构建）统一抽象为**接口契约**（见 `docs/standards/tool-contract.md`）。
- 前期全部用 **mock 实现**占位，返回预设数据；真实实现按同一契约**热替换**，不改调用方代码。
- 工具层**强制返回规模上限**（防大量数据撑爆 context）。
- 部分操作前期也可由老板**手工执行**，结果手动喂给 agent。

### 5. 人工 gate 策略

- **写库、调真实接口、SVN 提交**等敏感操作，必须显式人工确认后才执行。
- Agent 不得在未经人工确认的情况下自动触发上述操作。

### 6. 上下文控制策略

- 大数据（表数据、大日志）**不整体进 context**。
- Agent 带假设生成**针对性最小切片查询**（含 filter / limit / 聚合），只取验证所需数据。
- 大结果**落盘**，context 仅保留摘要 + 文件路径引用。

### 7. 隔离工作区策略（方向已定，细节待设计）

- 修复改动落到**独立工作区目录**（从目标项目主分支拉出），避免污染主工作副本。
- **开放问题**：构建依赖 IDEA 当前打开目录，隔离副本如何让构建作用到改动上——此问题留入 M2 启动前专项设计，不在 M0/M1 解决。

### 8. 构建策略

- 编译由**老板手工触发**（通过 IDEA 或命令行）。
- 编译报错可回灌给 agent 迭代（F4 编译反馈环）。
- 自动化构建不在当前范围内。

### 9. Agent 框架选型

- 选定 **LangChain Deep Agents（基于 LangGraph）**，内置 planning / ReAct / 子 agent / context 管理。
- 确认日期：2026-06-20。

## 理由

上述各点均为需求澄清阶段经充分讨论达成的共识，是所有后续设计的前提条件。固化在 ADR-0001，后续设计文档可直接引用，不必重新论证。

## 影响

- 所有后续设计以「mock 优先 + 统一契约」为基础；工具契约规范（`tool-contract.md`）需在 M1 mock 实现前完成。
- M1 实现需在 agent 框架选型确认后启动（或以最小可用实现占位并在 ADR 中记录选型）。
- 隔离工作区细节需在 M2（F3 应用修改）之前专项设计并记入新 ADR。
