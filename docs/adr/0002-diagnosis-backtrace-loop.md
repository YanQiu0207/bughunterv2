# ADR-0002：诊断采用「统一回溯循环」骨架

**状态**: 已接受
**日期**: 2026-06-19

## 背景

M1 诊断设计起步。借一个最小 NPE 例子（见 spec 4.4）厘清「诊断」的内在逻辑，确立 F1 的设计骨架。

## 决策

1. **核心洞察：抛出点 ≠ 根因点**。诊断的本质是「沿调用链回溯，找坏值的源头」；堆栈栈顶只是症状位置，根因常在调用链更深处或代码之外。

2. **F1 诊断设计为「统一回溯循环」**：定位嫌疑变量 → 回溯其来源 → 来源在代码内则继续读码、在代码外（DB/接口/配置）则调工具取证 → 直到坐实根因或线索耗尽 → 输出证据链。

3. **简单 bug = 退化情况，复杂 bug = 完整情况，共用同一循环**：故可从最简例子起步搭骨架，且骨架天然长到复杂场景，不白搭。

## 理由

统一模型让简单 / 复杂场景走同一条路，既满足「渐进式起步（铁律一）」，又保证骨架不会因起点太简单而失效。证据链作为循环的唯一产出，承接「可证伪」要求（见 ADR-0001 决策 3）。

## 影响

- M1 实现围绕「回溯循环」展开。
- mock 工具需支持「回溯到外部来源时的取证」这一关键动作。
- 下一步需设计：嫌疑变量定位策略、码内/码外判定、工具契约、上下文控制落点、状态/落盘格式。
